# ECDSA (Elliptic Curve Digital Signature Algorithm)

## 개념

타원곡선 위의 점 연산을 이용한 디지털 서명 알고리즘.
**개인키를 공개하지 않고도 메시지의 소유권을 증명**할 수 있다.
비트코인과 이더리움의 지갑 생성, 거래 서명에 사용되는 핵심 암호학.

```
앨리스 (보내는 사람)                    밥 (받는 사람) / 네트워크
─────────────────                    ──────────────────────
개인키 (비밀)                          공개키 (공개)
    │                                     │
    ▼                                     ▼
서명 생성 ──────── 서명 (r, s) ────────► 서명 검증
sign(개인키, 해시)                    verify(공개키, 해시, 서명)
    │                                     │
    ▼                                     ▼
"이 거래는 내가 했다"                "맞다, 앨리스가 한 거래다"

→ 개인키는 절대 네트워크에 노출되지 않음!
```

## 타원곡선이란?

### 곡선의 형태

```
y² = x³ + ax + b

    y │
      │       ╭─────╮
      │      ╱       ╲
      │    ╱           ╲
      │  ╭╯             ╲
──────┼─╯────────────────╲──── x
      │╲                 ╱
      │  ╰╮             ╱
      │    ╲           ╱
      │      ╲       ╱
      │       ╰─────╯

실수 위의 타원곡선은 이런 대칭 형태.
암호학에서는 유한체(mod p) 위에서 사용하므로
연속적인 곡선이 아닌 점들의 집합이 된다.
```

### 유한체 위의 타원곡선

```
실수 위:  연속적인 곡선 (무한한 점)
유한체 위: y² ≡ x³ + ax + b (mod p) 를 만족하는 정수 점들

예: y² = x³ + 2x + 3 (mod 97)

  이 곡선 위의 점들은 mod 97에서 방정식을 만족하는 (x, y) 쌍.
  예: (3, 6) → 6² = 36, 3³ + 2·3 + 3 = 36 → 36 ≡ 36 (mod 97) ✓
```

## 점 연산

### 점 덧셈 (Point Addition)

```
P ≠ Q 일 때, P + Q:

    y │
      │           P •
      │          ╱
      │         ╱
      │    Q • ╱
      │       ╱
──────┼──────╱────────•─── x
      │                R' (직선과 곡선의 세 번째 교점)
      │                │
      │                │ x축으로 반사
      │                ▼
      │                • P + Q = R

계산:
  기울기 λ = (y₂ - y₁) / (x₂ - x₁) mod p
  x₃ = λ² - x₁ - x₂ mod p
  y₃ = λ(x₁ - x₃) - y₁ mod p
```

### 점 더블링 (Point Doubling)

```
P = Q 일 때, 2P:

    y │
      │       P •──── 접선
      │      ╱   ╲
      │     ╱     ╲
      │    ╱       • R' (접선과 곡선의 교점)
──────┼───╱────────│─── x
      │            │ x축으로 반사
      │            ▼
      │            • 2P = R

계산:
  기울기 λ = (3x₁² + a) / (2y₁) mod p
  x₃ = λ² - 2x₁ mod p
  y₃ = λ(x₁ - x₃) - y₁ mod p
```

### 스칼라 곱 (Scalar Multiplication)

```
k × P = P + P + ... + P (k번)

이진 전개법 (Double-and-Add):
  예: 11P → 11 = 1011₂

  비트 순회:
    1 → result = P,     current = 2P
    1 → result = P+2P,  current = 4P    (= 3P)
    0 → result = 3P,    current = 8P
    1 → result = 3P+8P, current = 16P   (= 11P)

  → O(log k) 번의 연산으로 계산 가능!
  → 하지만 Q = kP에서 k를 역산하는 건 O(√p) → 사실상 불가능
```

## 키 생성

```
1. 랜덤 정수 d 선택 (1 ~ n-1)          ← 개인키 (비밀!)
2. Q = d × G 계산                       ← 공개키 (공개)

    개인키 d ──── 스칼라 곱 ────► 공개키 Q = dG
    (256비트 숫자)  (쉬움: O(256))   (곡선 위의 점)

    공개키 Q ──── ??? ────► 개인키 d
    (곡선 위의 점)  (불가능!)    역산 불가
                  ECDLP 문제

실제:
  개인키: 0x1a2b3c... (256비트 랜덤 정수)
  공개키: (0x79BE667E..., 0x483ADA77...) (곡선 위의 점)
  지갑 주소: RIPEMD160(SHA256(공개키)) → 1BvBMSEYstW...
```

## 서명 과정

### 서명 생성 (Sign)

```
입력: 개인키 d, 메시지 해시 e
출력: 서명 (r, s)

1. 랜덤 k 선택 (일회용 넌스, 1 ~ n-1)
2. R = k × G 계산
3. r = R.x mod n
4. s = k⁻¹ × (e + r × d) mod n
5. 서명 = (r, s)

┌─────────────────────────────────┐
│ 주의: k는 반드시 매번 새로운 랜덤! │
│                                 │
│ PS3 해킹 사건 (2010):            │
│ 소니가 ECDSA에서 k를 고정값으로    │
│ 사용 → 두 서명으로 개인키 역산됨!  │
│                                 │
│ k가 같으면: s₁ - s₂ 에서 d 계산 가능 │
└─────────────────────────────────┘
```

### 서명 검증 (Verify)

```
입력: 공개키 Q, 메시지 해시 e, 서명 (r, s)
출력: 유효/무효

1. s⁻¹ mod n 계산
2. u₁ = e × s⁻¹ mod n
3. u₂ = r × s⁻¹ mod n
4. R' = u₁ × G + u₂ × Q
5. R'.x mod n === r ?  → 유효!

수학적 증명:
  s = k⁻¹(e + rd) 에서
  k = s⁻¹(e + rd) = s⁻¹e + s⁻¹rd = u₁ + u₂d
  
  따라서:
  R' = u₁G + u₂Q = u₁G + u₂(dG) = (u₁ + u₂d)G = kG = R
  
  → R'.x = R.x = r  ✓ (올바른 서명이면 일치!)
```

## 왜 ECDSA인가?

### RSA vs ECDSA 보안 수준 비교

| 보안 강도 | RSA 키 크기 | **ECDSA 키 크기** | 비율 |
|---|---|---|---|
| 80비트 | 1024비트 | **160비트** | 6.4배 |
| 112비트 | 2048비트 | **224비트** | 9.1배 |
| **128비트** | **3072비트** | **256비트** | **12배** |
| 192비트 | 7680비트 | **384비트** | 20배 |
| 256비트 | 15360비트 | **512비트** | 30배 |

```
같은 보안 수준에서:
  RSA-3072:  서명 크기 384바이트, 키 크기 384바이트
  ECDSA-256: 서명 크기 64바이트,  키 크기 32바이트

→ 블록체인에서 매 거래마다 서명을 저장하므로 크기 차이가 매우 중요!
→ 비트코인은 secp256k1 (256비트) 사용 → 128비트 보안
```

## 블록체인에서의 사용

### 1. 지갑 주소 생성

```
비트코인:
  개인키 (256비트)
    │
    ▼ ECDSA 스칼라 곱 (secp256k1)
  공개키 (512비트 = x, y 좌표)
    │
    ▼ SHA-256
  해시1 (256비트)
    │
    ▼ RIPEMD-160
  해시2 (160비트)
    │
    ▼ Base58Check 인코딩
  비트코인 주소: 1BvBMSEYstWetqTFn5Au4m4GFg7xJaNVN2

이더리움:
  공개키 → Keccak-256 → 마지막 20바이트 → 0x 접두어
  → 0x71C7656EC7ab88b098defB751B7401B5f6d8976F
```

### 2. 거래 서명

```
앨리스 → 밥에게 1 BTC 전송:

1. 거래 데이터 구성:
   { from: 앨리스주소, to: 밥주소, amount: 1 BTC, nonce: 42 }

2. 거래 해시 계산:
   hash = SHA-256(거래 데이터)

3. ECDSA 서명:
   (r, s, v) = sign(앨리스_개인키, hash)
   v: 복구 ID (공개키 복원에 사용, 이더리움)

4. 서명된 거래 전파:
   { 거래 데이터, r, s, v }

5. 네트워크 검증:
   verify(앨리스_공개키, hash, (r, s)) → true → 블록에 포함
```

### 3. 소유권 증명

```
"이 지갑의 주인이 나다" 를 증명하는 방법:

  챌린지: "이 랜덤 메시지에 서명해보세요"
  응답:   sign(개인키, 챌린지 메시지) → 서명
  검증:   verify(공개키, 챌린지 메시지, 서명) → true

  → 개인키를 공개하지 않고도 소유권 증명 완료!
  → 이것이 "영지식 증명"의 기초적인 형태
```

```
비트코인/이더리움이 사용하는 곡선:

  곡선:   y² = x³ + 7
  소수 p: 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEFFFFFC2F
          (= 2²⁵⁶ - 2³² - 977)
  위수 n: 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141
  생성점 G:
    x: 0x79BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798
    y: 0x483ADA7726A3C4655DA4FBFC0E1108A8FD17B448A68554199C47D08FFB10D4B8

  → 이 구현에서는 학습용으로 작은 수(p=97)를 사용하지만,
    실제 비트코인은 위의 256비트 매개변수를 사용한다.
```

## 보안 고려사항

| 위협 | 대응 | 설명 |
|---|---|---|
| 개인키 유출 | 안전한 보관 | 하드웨어 지갑, 콜드 스토리지 |
| 넌스(k) 재사용 | 매번 새 랜덤 | 소니 PS3 해킹의 교훈 |
| 약한 난수 생성 | CSPRNG 사용 | 암호학적으로 안전한 난수 |
| 양자 컴퓨터 | 포스트양자 암호 | Shor 알고리즘으로 ECDLP 풀림 |
| 사이드 채널 공격 | 상수 시간 구현 | 타이밍 분석 방지 |

## 왜 알아야 하는가

ECDSA를 이해하면:
- 비트코인/이더리움 지갑이 어떻게 작동하는지 이해할 수 있다
- "개인키 잃으면 코인을 잃는다"의 수학적 이유를 알 수 있다
- 거래 서명이 왜 위조 불가능한지 이해할 수 있다
- 블록체인 보안 모델의 암호학적 기반을 파악할 수 있다
- 개인키를 공개하지 않고 소유권을 증명하는 원리를 이해할 수 있다
