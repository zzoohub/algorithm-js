# Gossip Protocol (가십 프로토콜)

## 개념

각 노드가 **무작위 소수의 피어에게** 정보를 전달하고,
그 피어들이 다시 **자기 피어에게** 전달하는 분산 통신 프로토콜.

마치 소문이 퍼지듯, 또는 전염병이 확산하듯 정보가 네트워크 전체로 퍼진다.

```
"나 새 거래 받았어" → 랜덤 3명에게 전달
                     → 그 3명이 각각 랜덤 3명에게 전달
                     → 그 9명이 각각 랜덤 3명에게 전달
                     → ... 네트워크 전체에 도달
```

## 라운드별 전파 (지수적 확산)

20개 노드 네트워크에서 fanout=3으로 전파하는 예시:

```
R0 (시작):  [●○○○○○○○○○○○○○○○○○○○]  1/20  (5%)
            시작 노드 1개만 메시지를 가짐

R1:         [●●●●○○○○○○○○○○○○○○○○]  4/20  (20%)
            시작 노드가 3개 피어에게 전달

R2:         [●●●●●●●●●●○○○○○○○○○○]  10/20 (50%)
            4개 노드가 각각 3개씩 → 최대 12개 새 노드
            (중복 제외하면 ~6개 추가)

R3:         [●●●●●●●●●●●●●●●●●●○○]  18/20 (90%)
            폭발적 확산, 대부분 도달

R4:         [●●●●●●●●●●●●●●●●●●●●]  20/20 (100%)
            전체 도달 완료

→ 20개 노드를 4라운드 만에 전부 커버
→ 각 노드는 최대 3개 연결만 사용 (효율적)
```

## 왜 가십인가

### 1. 확장성 (Scalability)
```
직접 브로드캐스트:
  한 노드가 N-1개 노드에게 직접 전송
  → 보내는 노드의 부하: O(N)
  → N=10,000이면 9,999개 연결 필요

가십:
  각 노드가 fanout개(보통 3~4)에게만 전달
  → 각 노드의 부하: O(fanout) = O(1)
  → N=10,000이어도 각 노드는 3~4개만 전달
```

### 2. 내결함성 (Fault Tolerance)
```
직접 브로드캐스트:
  보내는 노드가 죽으면 → 전파 실패 (단일 장애점)

가십:
  Node A → Node B, C, D
  Node B가 죽어도 → C, D가 계속 전파
  → 다중 경로로 전파되므로 일부 장애에 강함
  → 단일 장애점(Single Point of Failure)이 없음
```

### 3. 단순성 (Simplicity)
```
각 노드의 로직:
  1. 새 메시지를 받음
  2. 이미 아는 메시지인가? → 무시
  3. 새 메시지라면 → 랜덤 피어 몇 개에게 전달
  4. 끝

→ 글로벌 상태 관리 없음
→ 리더 선출 없음
→ 라우팅 테이블 없음
```

## 수렴 분석

### O(log N) 라운드 수렴

fanout = f일 때, 매 라운드마다 메시지를 아는 노드 수가 약 f배씩 증가:

```
R0: 1개 노드
R1: ~f개 노드
R2: ~f²개 노드
R3: ~f³개 노드
...
Rk: ~f^k개 노드 ≥ N  →  k ≈ log_f(N)

예: N=1000, f=3
   log₃(1000) ≈ 6.3 → 약 7라운드에 전체 도달

예: N=10,000, f=3
   log₃(10000) ≈ 8.4 → 약 9라운드에 전체 도달

→ 노드가 10배 늘어도 2~3라운드만 더 걸림!
```

### Fanout과 수렴 확률

```
fanout=1: 느리고 불안정 (일부 노드에 도달 못할 확률 높음)
fanout=2: 보통 수렴하지만 느림
fanout=3: 실용적 최솟값 (대부분의 시스템이 사용)
fanout=4+: 빠르지만 네트워크 부하 증가

fanout이 ln(N) 이상이면 높은 확률로 모든 노드에 도달:
  N=1000 → ln(1000) ≈ 6.9 → fanout=7이면 거의 확실히 수렴
  실무에서는 3~4로도 충분 (여러 라운드에 걸쳐 중복 전파되므로)
```

## 가십 변형: Push vs Pull vs Push-Pull

### Push 가십
```
메시지를 가진 노드가 랜덤 피어에게 "밀어넣기"

Node A (메시지 있음) → Node B: "이 메시지 받아"
                     → Node C: "이 메시지 받아"

장점: 새 메시지가 빠르게 퍼짐
단점: 후반부에 이미 아는 노드에게 중복 전송 (낭비)
```

### Pull 가십
```
메시지가 없는 노드가 랜덤 피어에게 "당겨오기"

Node X (메시지 없음) → Node A: "새 메시지 있어?"
                      ← Node A: "응, 여기 있어"

장점: 후반부에 효율적 (모르는 노드만 요청)
단점: 초반에 느림 (아무도 요청하지 않으면 전파 안 됨)
```

### Push-Pull 가십 (하이브리드)
```
초반: Push로 빠르게 확산
후반: Pull로 남은 노드 커버

1. Node A → Node B: "나 이런 메시지들 알아" (push)
2. Node B → Node A: "나는 이런 메시지들 알아" (pull)
3. 서로 모르는 메시지를 교환

→ 가장 효율적이며 실무에서 많이 사용
→ 비트코인은 Push-Pull 혼합 방식 사용
```

## 블록체인에서의 가십

### 비트코인 네트워크

비트코인 P2P 네트워크는 가십으로 트랜잭션과 블록을 전파한다.

```
트랜잭션 전파 (INV → GETDATA → TX):

1. Alice가 1 BTC를 Bob에게 보내는 트랜잭션 생성
2. Alice의 노드 → 연결된 피어에게 INV(inventory) 메시지:
   "txid: abc123... 알아?"
3. 모르는 피어 → GETDATA 요청: "abc123 보내줘"
4. Alice 노드 → TX 메시지로 실제 트랜잭션 전송
5. 받은 피어들도 자기 피어에게 INV → 반복

→ 수 초~수십 초 내에 전 세계 ~15,000 노드에 도달

블록 전파:
  동일한 방식이지만, 블록은 더 크므로
  Compact Block (BIP 152)로 최적화
  → 이미 가진 트랜잭션은 빼고 차이분만 전송
```

### 이더리움의 devp2p 프로토콜

이더리움은 devp2p라는 자체 P2P 프로토콜을 사용한다.

```
Discovery Protocol (노드 발견):
  Kademlia DHT 기반으로 피어를 발견
  → 부트노드에 접속 → 근처 노드 목록 획득 → 연결

Wire Protocol (메시지 교환):
  eth/68 프로토콜로 블록/트랜잭션 가십

트랜잭션 전파:
  - 연결된 피어의 √N 개에게 전체 트랜잭션 전송 (push)
  - 나머지 피어에게는 해시만 알림 (announce)
  - 해시만 받은 피어는 필요시 전체 데이터 요청 (pull)
  → Push-Pull 혼합으로 대역폭 절약

블록 전파:
  - 새 블록을 √N 피어에게 전체 전송
  - 나머지에게는 블록 해시만 알림
  - 필요시 블록 헤더 → 블록 바디 순서로 요청
```

## 실무 활용

### 1. 블록체인 P2P 네트워크
```
비트코인, 이더리움, 솔라나 등 모든 퍼블릭 블록체인이
가십으로 트랜잭션과 블록을 전파.

핵심 이유: 중앙 서버 없이 수만 개 노드가 동일한 정보를 공유해야 함
```

### 2. 전염병 프로토콜 (Epidemic Protocols)
```
Amazon DynamoDB, Riak:
  분산 DB에서 노드 간 데이터 동기화에 가십 사용
  → Anti-entropy: 주기적으로 피어와 데이터를 비교하여 불일치 해소
```

### 3. 장애 감지 (Failure Detection)
```
Cassandra의 Gossip-based Failure Detector:
  - 각 노드가 주기적으로 랜덤 피어에게 하트비트 전송
  - 하트비트가 오지 않는 노드 → 장애로 판정
  - 장애 정보도 가십으로 전파 → 모든 노드가 장애를 인지

HashiCorp Consul:
  - Serf 라이브러리 (SWIM 프로토콜 기반)
  - 멤버십 관리 + 장애 감지를 가십으로 수행
  - O(log N) 수렴으로 수천 노드에서도 빠르게 상태 전파
```

### 4. CockroachDB
```
클러스터 메타데이터 (노드 상태, 리더 정보 등)를 가십으로 전파.
→ 모든 노드가 클러스터 전체의 상태를 근사적으로 알 수 있음
```

## 비교: Gossip vs Flooding vs Structured Routing

| | Gossip | Flooding | Structured Routing |
|---|---|---|---|
| **방식** | 무작위 피어 일부에 전달 | 모든 피어에 전달 | DHT 등으로 특정 경로 전달 |
| **메시지 복잡도** | O(N log N) | O(N * E) | O(log N) |
| **수렴 시간** | O(log N) | O(직경) | O(log N) |
| **확실성** | 확률적 (높은 확률) | 확정적 | 확정적 |
| **내결함성** | 높음 | 높음 | 중간 (라우팅 깨지면 문제) |
| **대역폭** | 낮음 | 높음 (중복 많음) | 낮음 |
| **구현 복잡도** | 낮음 | 낮음 | 높음 (DHT 유지) |
| **사용처** | 블록체인 P2P, Cassandra | 긴급 알림, 소규모 네트워크 | IPFS, BitTorrent |

```
Flooding (플러딩):
  모든 피어에게 전달 → 빠르지만 메시지 폭발 (O(N*E))
  → 소규모 네트워크나 긴급한 경우에만 사용

Gossip (가십):
  랜덤 일부에게만 전달 → 약간 느리지만 효율적 (O(N log N))
  → 대규모 P2P 네트워크의 표준

Structured Routing (구조적 라우팅):
  DHT(분산 해시 테이블)로 정확한 경로 계산
  → 특정 노드에 도달하는 데 최적이지만, 브로드캐스트에는 부적합
  → IPFS의 파일 탐색, BitTorrent의 피어 발견에 사용
```

## 왜 알아야 하는가

가십 프로토콜을 이해하면:
- 블록체인 트랜잭션이 "어떻게" 네트워크에 퍼지는지 알 수 있다
- 분산 시스템에서 "리더 없이" 정보를 공유하는 원리를 이해할 수 있다
- 확장성과 내결함성의 트레이드오프를 체감할 수 있다
- Cassandra, Consul, CockroachDB 같은 시스템의 내부 동작을 이해할 수 있다
