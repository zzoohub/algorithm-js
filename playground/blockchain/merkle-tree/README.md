# Merkle Tree (머클 트리)

## 개념

거래(트랜잭션)들을 해시 쌍으로 묶어 올라가며 **하나의 루트 해시**를 만드는 이진 트리.
블록 안의 수천 개 거래를 단 하나의 해시로 요약하고, 특정 거래의 포함 여부를 **O(log n)**으로 증명한다.

```
"이 거래가 블록에 들어 있는가?" → 머클 트리가 답한다.
```

## 트리 구조

```
              ┌─────────────────┐
              │   머클 루트 (Root)  │  ← 블록 헤더에 저장되는 단 하나의 해시
              │   H(H_AB + H_CD)  │
              └────────┬────────┘
                 ┌─────┴─────┐
           ┌─────┴─────┐ ┌──┴──────┐
           │   H_AB    │ │   H_CD   │  ← 중간 노드
           │ H(H_A+H_B)│ │H(H_C+H_D)│
           └─────┬─────┘ └──┬──────┘
              ┌──┴──┐     ┌─┴───┐
           ┌──┴─┐ ┌─┴──┐ ┌┴──┐ ┌┴──┐
           │ H_A│ │ H_B│ │H_C│ │H_D│  ← 리프 노드: 각 거래의 해시
           └──┬─┘ └─┬──┘ └┬──┘ └┬──┘
              │     │     │     │
            Tx0   Tx1   Tx2   Tx3      ← 원본 거래 데이터
```

## 트리 빌드 과정

### 1단계: 리프 해시 생성
각 거래를 해시하여 리프 노드를 만든다.
```
Tx0 = "Alice→Bob 1BTC"    → H_A = hash("Alice→Bob 1BTC")
Tx1 = "Bob→Charlie 0.5BTC" → H_B = hash("Bob→Charlie 0.5BTC")
Tx2 = "Charlie→Dave 2BTC"  → H_C = hash("Charlie→Dave 2BTC")
Tx3 = "Dave→Eve 0.3BTC"    → H_D = hash("Dave→Eve 0.3BTC")
```

### 2단계: 쌍으로 묶어 해시
인접한 두 해시를 이어 붙여 새 해시를 만든다.
```
H_AB = hash(H_A + H_B)    ← 왼쪽 쌍
H_CD = hash(H_C + H_D)    ← 오른쪽 쌍
```

### 3단계: 루트까지 반복
```
Root = hash(H_AB + H_CD)  ← 모든 거래를 대표하는 단 하나의 해시
```

### 홀수 개일 때
거래가 홀수 개면 마지막 노드를 복제하여 짝을 맞춘다 (비트코인 방식).
```
[H_A, H_B, H_C]
→ [H_A, H_B, H_C, H_C]   ← H_C를 복제
→ [H_AB, H_CC]
→ [Root]
```

## 머클 증명 (Merkle Proof)

### 문제
"Tx2가 이 블록에 포함되어 있는가?"를 증명하고 싶다.
블록에 1000개의 거래가 있다면, 1000개를 전부 확인해야 할까?

### 답: O(log n)개의 해시만 있으면 된다

Tx2의 머클 증명에 필요한 해시 (★ 표시):
```
              ┌───────────┐
              │    Root    │  ← 3. 계산 결과와 비교
              └─────┬─────┘
               ┌────┴────┐
          ★┌───┴───┐ ┌───┴───┐
           │ H_AB  │ │ H_CD  │  ← 2. hash(H_C + H_D) 계산
           └───────┘ └───┬───┘
                      ┌──┴──┐
                   ┌──┴─┐ ★┌┴──┐
                   │ H_C│  │H_D│  ← 1. H_C = hash(Tx2), H_D는 증명에 포함
                   └──┬─┘  └───┘
                      │
                    [Tx2]             ← 검증하려는 거래
```

### 증명 경로
```
증명 = [
  { hash: H_D,  direction: "right" },   ← 형제 노드 (같은 레벨)
  { hash: H_AB, direction: "left"  },   ← 삼촌 노드 (한 레벨 위)
]
```

### 검증 과정
```
1. hash(Tx2)                  → H_C        (리프 해시)
2. hash(H_C + H_D)           → H_CD       (형제와 결합)
3. hash(H_AB + H_CD)         → Root'      (삼촌과 결합)
4. Root' === Root?            → true!      (거래가 포함되어 있음!)
```

## O(log n) 검증의 위력

| 블록 내 거래 수 | 전체 확인 시 필요한 해시 | 머클 증명에 필요한 해시 |
|---|---|---|
| 16 | 16 | 4 |
| 256 | 256 | 8 |
| 1,000 | 1,000 | ~10 |
| 1,000,000 | 1,000,000 | ~20 |

1000개 거래가 담긴 블록에서 1개 거래를 검증하려면:
- 전체 확인: 1000개 해시 계산 → **O(n)**
- 머클 증명: ~10개 해시 계산 → **O(log n)**

이 차이가 라이트 클라이언트를 가능하게 만든다.

## 실무에서의 활용

### 1. 비트코인 SPV (Simplified Payment Verification)
```
풀 노드:   블록 전체 (수백 MB)를 저장
라이트 노드: 블록 헤더 (80바이트)만 저장

라이트 노드가 거래를 검증하려면:
1. 풀 노드에게 머클 증명을 요청
2. 증명 (~10개 해시)을 받아서 루트까지 계산
3. 블록 헤더의 머클 루트와 비교 → 일치하면 거래 확인!

→ 스마트폰에서도 비트코인 거래를 검증할 수 있는 이유
```

### 2. Git의 트리 해싱
```
Git의 커밋 = 파일 트리의 머클 루트.
파일 하나가 바뀌면 → 해당 경로의 해시만 바뀜 → 루트 해시 변경.
→ 두 커밋의 루트 해시만 비교하면 변경 여부를 즉시 알 수 있다.
```

### 3. 인증서 투명성 (Certificate Transparency)
```
모든 SSL 인증서를 머클 트리에 기록.
누군가 위조 인증서를 발급하면 → 머클 루트가 달라짐 → 즉시 탐지.
Google Chrome이 이 방식으로 인증서 위조를 감시한다.
```

### 4. IPFS (InterPlanetary File System)
```
파일을 청크로 나누고, 각 청크의 해시를 머클 트리로 구성.
→ 루트 해시(CID)만 알면 파일 전체의 무결성을 검증 가능.
→ 일부 청크만 받아도 해당 부분이 올바른지 확인 가능.
```

### 5. 이더리움의 머클 패트리샤 트리
```
이더리움은 머클 트리를 확장한 Merkle Patricia Trie를 사용.
상태(잔액, 스토리지 등)를 트리에 저장하고,
상태 루트 해시를 블록 헤더에 포함.
→ 특정 계정의 잔액을 머클 증명으로 검증 가능.
```

## 핵심 요약

| 질문 | 답 |
|---|---|
| 머클 트리는 무엇? | 해시를 쌍으로 묶어 올라가는 이진 트리 |
| 루트 해시의 의미? | 모든 데이터의 요약 (1비트라도 바뀌면 루트가 변함) |
| 머클 증명이란? | 특정 데이터의 포함을 증명하는 해시 경로 |
| 왜 O(log n)? | 트리 높이만큼의 해시만 있으면 검증 가능 |
| 왜 중요? | 전체 데이터 없이도 부분 검증 → 라이트 클라이언트, SPV |
