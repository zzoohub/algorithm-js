# Count-Min Sketch (카운트-민 스케치)

## 개념

**"이 원소가 몇 번 나왔는가?"** 를 극도로 적은 메모리로 근사 추정하는 확률적 자료구조.
HashMap으로 정확히 세면 원소당 수십~수백 바이트가 필요하지만, Count-Min Sketch는 원소 수에 무관하게 **고정된 메모리**만 사용한다.

```
실시간 스트리밍 데이터에서:

HashMap 방식: {"apple": 5, "banana": 3, "cherry": 1, ...}
  → 100만 종류면 수십 MB

Count-Min Sketch: 1000×7 카운터 배열 = 56KB 고정
  → 100만 종류든 10억 종류든 동일
  → query("apple") = 5 또는 6 (약간 과대 추정 가능)
```

## 구조와 동작

### 2D 카운터 배열 (d행 × w열)

```
        열0  열1  열2  열3  열4
행0 h₀ [ 0 | 0 | 0 | 0 | 0 ]
행1 h₁ [ 0 | 0 | 0 | 0 | 0 ]
행2 h₂ [ 0 | 0 | 0 | 0 | 0 ]

각 행은 서로 다른 해시 함수 사용
```

### add("apple")
```
h₀("apple") % 5 = 2 → 행0[2] += 1
h₁("apple") % 5 = 0 → 행1[0] += 1
h₂("apple") % 5 = 4 → 행2[4] += 1

        열0  열1  열2  열3  열4
행0 h₀ [ 0 | 0 | 1 | 0 | 0 ]
행1 h₁ [ 1 | 0 | 0 | 0 | 0 ]
행2 h₂ [ 0 | 0 | 0 | 0 | 1 ]
```

### query("apple")
```
행0[2] = 1, 행1[0] = 1, 행2[4] = 1
min(1, 1, 1) = 1 ✓ 정확!
```

### 충돌이 일어나면?
```
add("banana") 후, h₁("banana") % 5 = 0이면 (충돌!)

행1 h₁ [ 2 | 0 | 0 | 0 | 0 ]  ← 행1[0]이 2가 됨

query("apple"):
행0[2] = 1, 행1[0] = 2, 행2[4] = 1
min(1, 2, 1) = 1 ✓ 최솟값이 충돌을 걸러냄!
```

**최솟값을 취하는 이유**: 충돌은 카운터를 올리기만 한다 (내리지 않는다). 따라서 d개의 행 중 충돌이 가장 적은 행의 값이 실제에 가장 가깝고, 최솟값이 그것이다.

## 오차 보장

Count-Min Sketch는 **수학적으로 증명된** 오차 범위를 보장한다:

```
추정값 ≤ 실제 빈도 + ε × N    (확률 ≥ 1 - δ)

여기서:
  ε = e / width    (상대 오차)
  δ = e^(-depth)   (오차 초과 확률)
  N = 전체 삽입 횟수
```

| 원하는 보장 | width | depth | 메모리 |
|------------|-------|-------|--------|
| 0.1% 오차, 99% 확률 | 2,719 | 5 | ~109KB |
| 0.01% 오차, 99.9% 확률 | 27,183 | 7 | ~1.5MB |

**한 방향 오차**: 과대 추정만 발생하고, 과소 추정은 절대 없다. `query(x) ≥ 실제 빈도(x)`.

## 확률적 자료구조 비교

| | Bloom Filter | HyperLogLog | Count-Min Sketch |
|---|---|---|---|
| 질문 | "있는가?" | "몇 종류?" | **"몇 번?"** |
| 오답 방향 | 거짓 긍정 | ±오차 | 과대 추정만 |
| 삭제 | 불가 | 불가 | 불가 |
| 병합 | OR | max | **합산** |

## 시간 복잡도

| 연산 | 복잡도 |
|------|--------|
| add | O(d) |
| query | O(d) |
| merge | O(d × w) |
| 공간 | O(d × w) |

d, w는 상수이므로 사실상 모든 연산이 O(1).

## 실무에서의 활용

### 1. 네트워크 트래픽 분석
초당 수백만 패킷이 흐르는 네트워크에서 "각 IP의 패킷 수"를 추적.
모든 IP를 HashMap에 넣을 수 없지만, Count-Min Sketch로 heavy hitter(트래픽 많은 IP)를 탐지.

### 2. 실시간 인기 검색어 (Top-K)
검색 쿼리 스트림에서 "현재 가장 인기 있는 검색어 10개"를 찾기.
Count-Min Sketch + Min-Heap 조합으로 O(1) 공간에서 Top-K를 유지.

### 3. 스트리밍 데이터 처리
Apache Spark, Flink 등의 스트림 프로세싱에서 윈도우별 빈도 추정.
메모리가 제한된 환경에서 무한 스트림을 처리.

### 4. 추천 시스템
"사용자가 이 카테고리의 상품을 몇 번 봤는가?"를 가볍게 추적.
정확한 수치보다 상대적 순위가 중요한 맥락에 적합.

### 5. CDN 캐시 정책
"이 URL이 몇 번 요청되었는가?" → 빈도가 높으면 캐시, 낮으면 무시.
모든 URL을 기억할 수 없을 때 Count-Min Sketch가 효율적.
