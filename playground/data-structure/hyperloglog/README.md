# HyperLogLog (초확률적 카디널리티 추정)

## 개념

**"고유 원소가 몇 개?"** 라는 질문에 **12KB의 메모리로** 10억 개를 세는 확률적 자료구조.
정확한 카운팅(HashSet)이 수 GB를 쓸 때, HyperLogLog는 ~2% 오차를 감수하고 메모리를 **99.99% 절약**한다.

```
정확한 방법 (HashSet):
  10억 개 고유 URL → ~40GB 메모리

HyperLogLog (precision=14):
  10억 개 고유 URL → ~12KB 메모리, 오차 ~0.81%
```

## 핵심 직관: 동전 던지기

**동전을 던져서 앞면이 연속 k번 나올 확률은 1/2^k.**
만약 누군가가 "연속 10번 앞면이 나왔어"라고 하면, "꽤 많이 던졌겠구나 (~1024번?)"라고 추측할 수 있다.

해시값의 이진 표현에서 연속 0의 수가 바로 이 "동전 던지기"에 해당한다:

```
hash("apple")  = 0001...  → 연속 0: 3개 → 2^3 = 8개 추정
hash("banana") = 0000001... → 연속 0: 6개 → 2^6 = 64개 추정

관측된 최대 연속 0 = max(3, 6) = 6
→ 대략 2^6 = 64개의 고유 원소가 있다고 추정
```

### 문제: 분산이 너무 크다
하나의 관측으로는 운이 좋거나 나쁜 경우에 크게 틀린다.

### 해결: 확률적 분할 + 조화평균

```
해시값을 m개의 버킷으로 나눔 (상위 비트로 분류)

hash = 1010 | 000101...
       ^^^^   ^^^^^^^^^
     버킷 번호  이 부분에서 연속 0을 카운트
     (10번)

각 버킷에서 관측된 최대 연속 0을 저장:
  버킷[0] = 3  → 2^3
  버킷[1] = 5  → 2^5
  버킷[2] = 2  → 2^2
  ...

조화평균: m / (2^(-3) + 2^(-5) + 2^(-2) + ...) × 보정상수
```

**왜 조화평균?** 산술평균은 이상치에 민감하지만, 조화평균은 큰 값의 영향을 줄여서 더 안정적인 추정을 준다.

## 정밀도와 메모리 트레이드오프

| Precision (p) | 레지스터 수 (m) | 메모리 | 표준 오차 |
|---|---|---|---|
| 4 | 16 | 16 bytes | 26% |
| 10 | 1,024 | 1 KB | 3.25% |
| 14 | 16,384 | 12 KB | **0.81%** |
| 16 | 65,536 | 48 KB | 0.41% |

Redis의 HyperLogLog는 p=14를 사용 (12KB, 0.81% 오차).

## 병합 (Union)

HyperLogLog의 강력한 특성: **두 HLL을 O(m)에 병합**할 수 있다.
각 레지스터의 최댓값을 취하면 합집합의 카디널리티를 추정.

```
서버 A의 방문자 HLL + 서버 B의 방문자 HLL
= 전체 고유 방문자 HLL

merge: registers[i] = max(A.registers[i], B.registers[i])
```

이 성질 덕분에 분산 환경에서 각 노드가 독립적으로 HLL을 유지하고, 나중에 합칠 수 있다.

## 시간 복잡도

| 연산 | 복잡도 |
|------|--------|
| add | O(1) |
| count | O(m) |
| merge | O(m) |
| 메모리 | O(m) = O(2^p) |

## 실무에서의 활용

### 1. Redis PFCOUNT
Redis의 `PFADD`, `PFCOUNT`, `PFMERGE` 명령어가 바로 HyperLogLog.
하루 UV(Unique Visitor) 수를 12KB로 추적한다.

```
PFADD visitors:2024-01-15 "user:123" "user:456" ...
PFCOUNT visitors:2024-01-15
→ 약 1,234,567 (0.81% 오차)
```

### 2. 대규모 로그 분석
"오늘 하루 고유 IP 수", "고유 검색 쿼리 수", "고유 에러 메시지 수" 등
정확한 카운트가 필요 없고 규모 파악만 하면 되는 대부분의 분석.

### 3. 데이터베이스 쿼리 최적화
PostgreSQL, BigQuery 등이 `COUNT(DISTINCT column)`의 내부 추정에 HLL 변형을 사용.
쿼리 플래너가 조인 전략을 결정할 때 카디널리티 추정이 핵심.

### 4. 네트워크 모니터링
DDoS 탐지: "지난 1분간 고유 소스 IP 수"가 급증하면 공격 의심.
각 시간 윈도우마다 HLL을 유지하고, 카디널리티 변화를 모니터링.

### 5. A/B 테스트 인프라
실험 그룹별 고유 사용자 수를 실시간으로 추적.
정확한 수치가 아니라 "대략적 규모"와 "추세"가 중요한 맥락.

### 왜 Bloom Filter와 다른가?
- Bloom Filter: "이 원소가 집합에 있는가?" (멤버십 테스트)
- HyperLogLog: "집합에 원소가 총 몇 개인가?" (카디널리티 추정)
둘 다 확률적이지만 답하는 질문이 다르다.
