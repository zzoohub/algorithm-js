# Consistent Hashing (일관된 해싱)

## 개념

분산 시스템에서 **"데이터를 어떤 서버에 저장할 것인가?"** 를 결정하는 알고리즘.
서버가 추가되거나 제거될 때 **최소한의 데이터만 재배치**한다.

### 일반 해싱의 문제

```
server = hash(key) % 3    (서버 3대)

hash("user:1") % 3 = 0  → 서버 A
hash("user:2") % 3 = 1  → 서버 B
hash("user:3") % 3 = 2  → 서버 C

서버 1대 추가 (3대 → 4대):
hash("user:1") % 4 = 1  → 서버 B로 이동!  ← 바뀜
hash("user:2") % 4 = 2  → 서버 C로 이동!  ← 바뀜
hash("user:3") % 4 = 0  → 서버 A로 이동!  ← 바뀜

→ 거의 모든 데이터가 재배치됨 (약 75%)
→ 캐시 미스 폭발, 대규모 데이터 마이그레이션
```

### Consistent Hashing의 해법

```
해시 공간을 원(ring)으로 만들어서, 서버와 키를 모두 원 위에 배치.
각 키는 시계방향으로 가장 가까운 서버에 할당.

         0°
      A . . B
     .         .
   .             .
 270°               90°
   .             .
     .         .
      C . . .
        180°

key1(hash=40°)  → 시계방향 → B(70°)   → 서버 B
key2(hash=150°) → 시계방향 → C(200°)  → 서버 C
key3(hash=280°) → 시계방향 → A(350°)  → 서버 A

서버 D를 100°에 추가하면:
→ 70°~100° 사이의 키만 B→D로 이동
→ 나머지는 전혀 영향 없음!
→ 평균 K/N개의 키만 이동 (K=전체 키 수, N=서버 수)
```

## 가상 노드 (Virtual Nodes)

### 문제: 불균등 분포
물리 서버가 3대뿐이면 원 위의 위치가 불균등해서 특정 서버에 부하가 집중될 수 있다.

### 해결: 가상 노드
각 물리 서버를 원 위의 여러 지점에 배치.

```
물리 서버 A, B만 있을 때:

가상 노드 없이:           가상 노드 150개씩:
     A                    A₁ B₃ A₅ B₁
    . .                  . . . . . .
  .     B              A₃ B₂ A₂ B₅ A₄
  → A가 75%, B가 25%    → 거의 50:50
```

가상 노드 수가 많을수록 분포가 균등해진다. 실무에서는 100~200개가 적절.

## 재배치 비율 비교

| 방법 | 서버 추가 시 재배치 비율 |
|------|--------------------------|
| hash % N | ~(N-1)/N (거의 전부) |
| **Consistent Hashing** | **~1/N (최소한)** |

서버 10대에서 11대로 늘릴 때:
- 일반 해싱: ~91%의 키가 이동
- Consistent Hashing: ~9%의 키만 이동

## 시간 복잡도

| 연산 | 복잡도 |
|------|--------|
| 키 → 노드 조회 (getNode) | O(log(V×N)) 이진 탐색 |
| 노드 추가 (addNode) | O(V × log(V×N)) |
| 노드 제거 (removeNode) | O(V) |
| 공간 | O(V × N) |

V = 가상 노드 수, N = 물리 노드 수

## 실무에서의 활용

### 1. DynamoDB / Cassandra
파티션 키를 consistent hash로 노드에 분배. 노드 추가/제거 시 최소 데이터만 재배치.
Cassandra의 `partitioner`가 바로 consistent hashing.

### 2. CDN (Content Delivery Network)
"이 URL의 콘텐츠를 어떤 엣지 서버에 캐시할 것인가?"
서버가 장애나면 해당 서버의 키만 다음 서버로 넘어감. 나머지 캐시는 유지.

### 3. 로드 밸런서
Nginx, HAProxy의 consistent hashing 모드.
같은 사용자의 요청을 항상 같은 백엔드 서버로 보내 세션 어피니티 유지.

### 4. 분산 캐시 (Memcached, Redis Cluster)
캐시 키를 여러 노드에 분배. 노드가 죽어도 나머지 캐시는 살아있음.
Redis Cluster는 16384개의 해시 슬롯을 사용하는 변형.

### 5. 객체 스토리지 (S3, GCS)
파일을 여러 스토리지 노드에 분산 저장. replication factor와 함께 사용하면 `getNodes(key, 3)`으로 3개의 복제 위치를 결정.

### 왜 개발자가 알아야 하는가
분산 시스템 면접의 단골 주제이기도 하지만, 더 중요한 건 "상태를 가진 분산 시스템에서 스케일링할 때 데이터 이동을 최소화한다"는 사고 방식이 아키텍처 설계 전반에 적용된다.
