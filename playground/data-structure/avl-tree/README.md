# AVL Tree (자가 균형 이진 탐색 트리)

## 개념

BST(이진 탐색 트리)의 치명적 문제를 해결하는 자가 균형 트리.
**"어떤 순서로 데이터를 넣든 항상 O(log n)을 보장한다."**

### BST의 문제

```
1, 2, 3, 4, 5를 순서대로 BST에 넣으면:

1                   ← 사실상 연결 리스트
 \                     탐색 O(n)
  2
   \
    3
     \
      4
       \
        5
```

### AVL의 해법

```
같은 데이터를 AVL에 넣으면:

       2
      / \
     1   4            ← 항상 균형
        / \              탐색 O(log n)
       3   5
```

## 핵심 원리: Balance Factor

모든 노드에서 **왼쪽 서브트리 높이 - 오른쪽 서브트리 높이**가 `-1, 0, 1` 중 하나여야 한다.
이 불변량이 깨지면 **회전(rotation)**으로 복구한다.

```
BF = height(left) - height(right)

  [3]  BF = 2-1 = 1  ✓ 균형
  / \
 [2] [1]

  [3]  BF = 3-1 = 2  ✗ 불균형 → 회전 필요!
  /
 [2]
 /
[1]
```

## 4가지 회전

### LL (Left-Left) → 오른쪽 회전 1회
왼쪽 자식의 왼쪽에 삽입되어 불균형.
```
      z (+2)            y (0)
     / \              /    \
    y   T4    →     x       z
   / \             / \     / \
  x   T3         T1  T2  T3  T4
 / \
T1  T2
```

### RR (Right-Right) → 왼쪽 회전 1회
오른쪽 자식의 오른쪽에 삽입되어 불균형.
```
  z (-2)                y (0)
 / \                  /    \
T1   y        →     z       x
    / \            / \     / \
   T2   x        T1  T2  T3  T4
       / \
      T3  T4
```

### LR (Left-Right) → 왼쪽 회전 + 오른쪽 회전
왼쪽 자식의 오른쪽에 삽입. 한 번의 회전으로는 안 되고 두 번 필요.
```
    z                z                 x
   / \              / \              /   \
  y   T4    →     x   T4    →     y       z
 / \             / \             / \     / \
T1   x         y   T3         T1  T2  T3  T4
    / \       / \
   T2  T3   T1  T2
```

### RL (Right-Left) → 오른쪽 회전 + 왼쪽 회전
오른쪽 자식의 왼쪽에 삽입. LR의 대칭.

## 시간 복잡도

| 연산 | BST (최악) | AVL Tree |
|------|------------|----------|
| 탐색 | O(n) | **O(log n)** |
| 삽입 | O(n) | **O(log n)** |
| 삭제 | O(n) | **O(log n)** |
| 공간 | O(n) | O(n) |

삽입/삭제 시 최대 O(log n)번의 회전이 발생하지만, 각 회전은 O(1)이므로 전체는 여전히 O(log n).

## AVL vs Red-Black Tree

| | AVL | Red-Black |
|---|---|---|
| 균형 조건 | 높이 차 ≤ 1 (엄격) | 색상 규칙 (느슨) |
| 탐색 | 약간 더 빠름 (더 균형적) | 약간 느림 |
| 삽입/삭제 | 회전 더 많음 | 회전 적음 |
| 사용처 | DB 인덱스, 읽기 많은 곳 | 언어 표준 라이브러리 |

- Java TreeMap, C++ std::map → Red-Black Tree (삽입/삭제가 잦아서)
- DB 인덱스, 메모리 DB → AVL이 유리할 수 있음 (읽기가 압도적이라서)

## 실무에서의 활용

### 1. 데이터베이스 인덱스
인메모리 DB의 인덱스. 디스크 기반에서는 B-Tree가 더 적합하지만,
메모리에서는 AVL/Red-Black이 캐시 효율적이다.

### 2. 자동완성 / 사전
정렬된 데이터에 대한 빠른 범위 검색.
"abc"로 시작하는 모든 단어를 O(log n + k)에 찾기.

### 3. 스케줄러
우선순위가 있는 작업을 관리. 삽입/삭제/최솟값 찾기가 모두 O(log n).
Linux의 CFS(Completely Fair Scheduler)는 Red-Black Tree를 사용.

### 4. 기하 알고리즘의 상태 관리
Sweep Line 알고리즘에서 현재 활성 세그먼트들을 균형 트리로 관리.
삽입/삭제/이웃 찾기가 빈번한 환경.

### 5. 왜 이해해야 하는가
자가 균형 트리의 핵심 사고 — **"불변량을 깨뜨리는 변화가 올 때, 최소한의 구조 변경으로 불변량을 복구한다"** — 는 분산 시스템의 리밸런싱, 로드 밸런서의 가중치 조정 등 더 넓은 맥락에서도 동일하게 적용된다.
